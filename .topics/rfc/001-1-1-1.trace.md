## Purpose

This iteration focuses on examples rather than restating the full
envelope argument.

The goal is to show how one envelope model can cover:
- pointer artifacts
- text artifacts
- markdown artifacts
- Tiinex Provenance runtime exports
- Tiinex Feedback runtime artifacts

---

## Example 1: Pointer Trace

This example shows a body-less `.trace.md` that acts like a link.

It keeps a local parent trace for lineage continuity while using a
stable committed origin to point at another artifact.

    # Continuity Context

    - Parent Trace: [001-1-1.trace.md](001-1-1.trace.md)

---

## Design Reading

Taken together, the examples suggest that the envelope model should
stay lightweight and let schema names do most of the differentiation.

The important variation is not whether the file ends with `.md` or
`.trace.md`, but what kind of body or no-body pattern the artifact is
carrying.

That means the current model can reasonably support:
- envelope-only pointer traces
- prose RFC traces
- runtime evidence traces
- topic-centered feedback artifacts

without forcing one universal body shape across all of them.

---

## Conclusion

The current example set is already enough to demonstrate a useful
continuity family:

1. pointer
2. text
3. markdown
4. provenance runtime
5. feedback runtime

The main remaining design choice for feedback is not whether lineage is
possible, but whether it needs its own exported trace artifact yet.
