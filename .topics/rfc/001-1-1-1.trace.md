## Purpose

This iteration focuses on examples rather than restating the full
envelope argument.

The goal is to show how one envelope model can cover:
- pointer artifacts
- text artifacts
- markdown artifacts
- Tiinex Provenance runtime exports
- Tiinex Feedback runtime artifacts

This iteration also adds a minimum reference model:
- one `Parent Trace`
- one `Parent-Integrity`
- optional `Reference Traces`
- optional `Reference-Integrities`

## Design Reading

Taken together, the examples suggest that the envelope model should
stay lightweight and let schema names do most of the differentiation.

The important variation is not whether the file ends with `.md` or
`.trace.md`, but what kind of body or no-body pattern the artifact is
carrying.

That means the current model can reasonably support:
- envelope-only pointer traces
- prose RFC traces
- runtime evidence traces
- topic-centered feedback artifacts

The minimum reference model also lets an artifact stay in one primary
lineage while carrying checksum-protected side references to other
relevant traces.

without forcing one universal body shape across all of them.

## Suggested Integrity Procedure

For repository-backed lineage, the conservative integrity procedure is
sequential rather than batch-oriented.

Suggested order:

1. commit the parent artifact
2. read the committed parent body only, excluding the `Continuity Integrity` footer
3. compute the parent's `Body-Integrity`
4. write that value into the child's `Parent-Integrity`
5. compute the child's own `Body-Integrity`
6. commit the child artifact
7. continue forward through the lineage

This procedure is useful because each child can refer to an already
locked parent body rather than to an uncommitted workspace state.

For references, the same principle applies:

- `Reference-Integrities` should ideally be copied from already
  committed reference artifacts
- footer values should be written after the body content is settled
- integrity footers should remain excluded from body hashing so footer
  updates do not recursively invalidate the body hash

This document treats that sequence as the preferred operational pattern
for achieving stable lineage integrity in a Git-backed workflow.

## Advanced Origin Integrity Profile

The minimum model is sufficient for local lineage integrity, but it is
not always sufficient to validate a remote or external `Origin` with
the same strength.

For that stronger case, this document recommends an optional advanced
profile rather than a heavier default model.

Candidate profile name:

- `origin-lineage-chain-sha256-v1`

Suggested meaning:

1. start at the artifact identified by `Origin`
2. follow parent lineage backward while each step remains inside the
  same origin repository or origin source context
3. collect the committed `Body-Integrity` value for each step in order
4. serialize that ordered chain in a canonical format
5. compute one SHA-256 over the serialized chain

This profile should be treated as stronger than direct
`Parent-Integrity`, but also more expensive and more conditional.

It is most useful when:

- `Origin` points outside the local repository or filesystem context
- stronger historical verification is needed
- the origin repository itself carries a meaningful lineage chain

It should remain optional because many continuity cases only require
direct local parent integrity and do not benefit from a full origin
chain walk.

This document does not recommend raw hash concatenation without a
canonical serialization rule. The chain should be versioned and
serialized deterministically before hashing.

---

## Conclusion

The current example set is already enough to demonstrate a useful
continuity family:

1. pointer
2. text
3. markdown
4. provenance runtime
5. feedback runtime

The minimum next-step relation model is:

1. `Parent Trace`
2. `Parent-Integrity`
3. `Reference Traces`
4. `Reference-Integrities`

The preferred operational integrity pattern is sequential:
lock the parent, copy its committed integrity forward, then lock the
child.

When direct parent integrity is not enough for an external origin,
`origin-lineage-chain-sha256-v1` is a reasonable optional stronger
profile.

The least stable part of the set is feedback, where the artifact shape
exists but the lineage strategy is still unresolved.
