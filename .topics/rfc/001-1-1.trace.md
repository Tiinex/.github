                Discrimination-First Continuity Envelope

Status of This Memo

  This memo provides information for the Tiinex continuity ecosystem.

  This document does not define an Internet standard of any kind.
  Distribution of this memo is unlimited.

Abstract

This document defines a discrimination-first continuity envelope for
human-readable trace artifacts.

The envelope is designed to help both humans and runtimes determine,
quickly and with bounded effort, whether a continuity artifact is:
- relevant
- reconstructable
- compatible
- worth deeper inspection

The design intentionally prioritizes:
- continuity orientation
- discriminating metadata
- recoverable lineage
- runtime agnosticism
- human-readable structure

over:
- full serialization metadata
- dense machine-first headers
- hidden runtime coupling
- hardcoded origin systems

---

1.  Introduction

RFC-0001 defines the continuity problem space explored by the Tiinex
project. RFC-0002 introduced an experimental trace continuity
structure.

This document is the next iteration of that work.

It narrows the header model to the fields with the highest orientation
value and separates four concerns that should not be collapsed into a
single identifier:
- semantic continuity relationship
- reconstruction entrypoint
- raw content interpretation
- integrity boundary

---

2.  Design Goal

A continuity envelope SHOULD let a human or runtime answer the
following question without reading the entire document body:

> Is this artifact part of the continuity chain I care about, and can I
> reconstruct it with the capabilities I currently have?

The envelope therefore optimizes for:
- discrimination-first reading
- bounded reconstruction decisions
- explicit raw-data description
- portable future adapters

The envelope does not attempt to provide a complete execution model or
canonical serialization surface.

---

3.  Envelope Structure

A continuity envelope SHOULD contain:
- a lightweight header
- an unchanged document body
- an integrity footer

The header exists for orientation.

The body remains the primary narrative and specification surface.

The footer exists for integrity and bounded validation.

---

4.  Continuity Context Header

The recommended header ordering is:

1. Parent Trace
2. Origin
3. Type
4. Schema

This ordering intentionally places the highest continuity
discrimination value first.

Example:

    # Continuity Context

    - Parent Trace: [001-1.trace.md](001-1.trace.md)
    - Origin:
      git://external-rfc-snapshot@8f2a91e:rfc/001.trace.txt
    - Type: markdown
    - Schema: tiinex.rfc.trace-continuity-draft.v1

---

5.  Parent Trace

The Parent Trace field identifies the immediate semantic continuity
ancestor.

It answers:
- what continuity chain this artifact belongs to
- which prior step it most directly continues

Parent Trace is a logical continuity relation. It is not a storage
locator and it is not an integrity proof.

---

6.  Origin

The Origin field identifies the best known bounded reconstruction
entrypoint.

For a lineage root, Origin MAY be omitted when any explicit locator
would only point back to the root artifact itself.

For non-root artifacts, Origin SHOULD identify the inherited recovery
entrypoint carried forward from the parent continuity step when such an
entrypoint exists. It is not the current artifact's own path.

Origin is only needed when that inherited recovery entrypoint cannot be
assumed from the local repository or filesystem context, or when the
parent continuity step lives outside that local context.

When the source should be disclosed, Origin SHOULD use an
adapter-prefixed locator form:

  adapter-name://...

The adapter prefix identifies the reconstruction strategy, not the
continuity identity.

Examples include:

  git://github.com/Tiinex/.github@1e6762cc97ef151828fa7466ec22e7a2f1762909:.topics/rfc/001.trace.txt
  https://github.com/Tiinex/.github/blob/1e6762cc97ef151828fa7466ec22e7a2f1762909/.topics/rfc/001.trace.txt
  archive://rfc-snapshot.zip/.topics/rfc/001.trace.txt
  file://C:/continuity/rfc/001.trace.txt

When the source should not be disclosed, or when the original locator
has been intentionally removed from the artifact, the value
`REDACTED` is valid.

Unknown adapters do not invalidate the artifact. They only limit
direct reconstruction by runtimes that do not support them.

---

7.  Content Description

The header SHOULD describe the raw body content directly.

Type identifies the body form in lightweight, human-readable terms.

Schema identifies the body structure with a stable,
implementation-defined schema name.

These fields exist to reduce guesswork. They are preferred over
meta-reasoning fields such as compatibility claims, confidence labels,
or resolution summaries when the goal is bounded reconstruction of raw
data.

---

8.  Footer Integrity

The footer SHOULD be placed at the bottom of the artifact so that it
does not interrupt continuity orientation or narrative reading.

The footer is the appropriate location for:
- body checksums
- integrity policies
- compact validation anchors

The footer SHOULD carry:
- `Integrity Policy`
- `Body-Integrity`
- `Parent-Integrity` when a prior lineage step exists

The footer SHOULD be excluded from its own integrity calculation.

---

9.  Deliberate Exclusions

The header SHOULD NOT begin with:
- opaque database identifiers
- serialization internals
- protocol dumps
- parser-specific state
- checksum-first ordering

Such fields may be useful elsewhere, but they do not carry the highest
orientation value for continuity reading.

---

10.  Publication Direction

This document is written as a markdown continuity artifact, not as a
final Internet-Draft submission format.

For eventual publication, the content should be converted into the
required Internet-Draft authoring and stream process used by the RFC
Editor ecosystem.

The intended contribution of this iteration is to stabilize the model
before transport and publication mechanics are finalized.

---

11.  Security Considerations

Origin fields, lineage references, and content descriptors may expose
historical locations, infrastructure details, or operational context.

Implementations SHOULD support omission, redaction, or bounded
disclosure where appropriate.

---

12.  IANA Considerations

This document has no IANA actions.

---

13.  Conclusion

The discrimination-first continuity envelope preserves what matters
most at the start of reading:
- chain membership
- reconstruction entrypoint
- raw content interpretation
- recoverable orientation

By separating Parent Trace, Origin, and footer integrity into distinct
responsibilities, the model improves continuity portability without
overloading any one field.
