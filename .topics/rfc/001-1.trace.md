# RFC-0002 — Trace Continuity Structure (Draft)

Status: Experimental
Category: Trace Continuity
Project: Tiinex Continuity RFCs

---

# 1. Abstract

This document defines an experimental human-readable continuity trace structure.

The goal is not creating immutable truth systems.

The goal is enabling:
- recoverable continuity
- bounded reconstruction
- runtime discrimination
- inspectable lineage
- portable continuity artifacts

This RFC intentionally prioritizes:
- readability
- continuity orientation
- human inspection
- lightweight parsing
- runtime agnosticism

over:
- rigid serialization
- protocol density
- machine-first formatting

---

# 2. Design Philosophy

Trace files SHOULD remain:
- readable during extended inspection
- navigable without tooling
- understandable in plain markdown
- operationally lightweight

A human reader SHOULD be able to:
- determine relevance quickly
- understand lineage relationships
- identify runtime compatibility
- locate reconstruction anchors
- continue work manually if necessary

---

# 3. Continuity Context

Trace files SHOULD begin with a lightweight continuity context section.

The purpose of the section is:
- continuity discrimination
- reconstruction orientation
- content identification

The section SHOULD remain:
- human-readable
- markdown-native
- visually lightweight

Example:

```md
# Continuity Context

- Parent Trace:
  [001-1-sigma.trace.md](../001-1-sigma.trace.md)

- Origin:
  git://github.com/Tiinex/rfcs@8f2a91e:rfc-0002/001.trace.txt

- Type:
  markdown

- Schema:
  tiinex.rfc.trace-continuity-draft.v1
```

---

# 4. Parent Trace

The Parent Trace field represents the logical continuity parent.

The field SHOULD identify:
- the immediate continuity ancestor
- the prior reconstructable step
- the semantic continuity relationship

The Parent Trace field does not require:
- Git
- centralized infrastructure
- canonical hosting providers

---

# 5. Origin

The Origin field represents the reconstruction entrypoint.

For non-root artifacts, Origin refers to the inherited reconstruction
entrypoint associated with the parent continuity step. It does not
identify the current artifact's own file location.

Origin is only needed when that inherited recovery entrypoint cannot be
assumed from the local repository or filesystem context, or when the
parent continuity step lives outside that local context.

The Origin SHOULD resolve directly to:
- the trace artifact
- the historical snapshot
- the reconstructable source location

The Origin MAY reference:
- Git repositories
- local filesystems
- network storage
- archives
- remote continuity systems
- future runtime-specific origins

Examples MAY include:

```md
- Origin:
  https://github.com/example/project/blob/abc123/trace.md
```

```md
- Origin:
  file://archive/trace-history/001-1.trace.md
```

```md
- Origin:
  smb://continuity-server/projects/rfc-0002/001.trace.md
```

The continuity model intentionally avoids hardcoding Git as the only valid origin system.

When the parent trace is already locally resolvable inside the same
repository or filesystem context, Origin MAY be omitted.

---

# 6. Content Identification

Content fields exist to help both:
- humans
- runtimes

determine whether a trace is:
- relevant
- reconstructable
- directly interpretable as raw data

The format intentionally favors:
- discrimination
- content identification
- bounded reconstruction

over:
- runtime-specific inference

When the source location should not be disclosed, `Origin: REDACTED`
is valid and preferred over silent omission or a misleading locator.

---

# 7. Human-Readable Metadata

Metadata SHOULD remain:
- soft structured
- markdown-native
- visually calm
- minimally overwhelming

The format intentionally avoids:
- dense serialization dumps
- excessive protocol syntax
- machine-first visual design

Markdown lists are preferred because they:
- preserve readability
- reduce visual overload
- support continuity inspection
- remain naturally diffable

---

# 8. Integrity Footer

Trace files MAY include an integrity footer.

The integrity footer SHOULD appear at the bottom of the trace.

The footer exists to:
- support lineage consistency checks
- support reconstruction confidence
- support compact validation
- support bounded integrity verification

The footer SHOULD carry:
- `Integrity Policy`
- `Body-Integrity`
- `Parent-Integrity` when a prior lineage step exists

Example:

```md
---

- Integrity Policy: parent-lineage-sha256-v1
- Body-Integrity: sha256:3b6e...
- Parent-Integrity: sha256:a91f...
```

---

# 9. Footer Exclusion Rule

Integrity calculations SHOULD exclude the integrity footer region itself.

The footer is treated as:
- removable
- replaceable
- recalculable

This avoids recursive self-hashing problems.

---

# 10. Integrity Policies

This RFC intentionally does not standardize:
- traversal depth
- recursive hashing behavior
- compact strategies
- lineage replay algorithms

Instead, traces MAY declare integrity policies.

Example:

```md
- Integrity Policy:
  parent-lineage-sha256-v1
```

Under `parent-lineage-sha256-v1`, `Body-Integrity` is the SHA-256 of
the artifact body with the integrity footer excluded. `Parent-Integrity`
is the parent's `Body-Integrity` value when a prior lineage step exists.

Tooling MAY implement:
- different traversal models
- different compact strategies
- different reconstruction policies

while remaining continuity-compatible.

---

# 11. Recoverability Over Immutability

The trace model prioritizes:
- recoverability
- reconstruction
- portability
- inspectability

The goal is not:
- permanent immutable truth
- global canonical consensus
- blockchain-style verification

Integrity systems exist to support:
continuity consistency.

---

# 12. Runtime Agnosticism

The trace structure SHOULD remain usable across:
- local runtimes
- cloud runtimes
- editor integrations
- archive systems
- future continuity tooling

No runtime is treated as permanently canonical.

---

# 13. Compact Compatibility

Trace structures SHOULD support future compact operations.

Compact behavior MAY:
- summarize lineage
- archive continuity
- replace historical detail
- preserve reconstruction anchors

The continuity model assumes:
full historical state may not always remain locally present.

Recoverability is prioritized over permanent duplication.

---

# 14. Human Continuity

The trace format prioritizes preserving:
- orientation
- readability
- continuity flow
- inspectable evolution

A trace SHOULD remain comfortable for humans to inspect during prolonged reading.

Human continuity degradation is treated as a real operational problem.

This RFC is experimental and subject to revision.

The format intentionally remains flexible while:
- runtime behaviors evolve

- reconstruction strategies improve

The trace structure is intended to function as:
- a continuity artifact

# 6. Parent Trace

The Parent Trace field identifies the immediate semantic continuity
ancestor.

It answers:
- what continuity chain this artifact belongs to
- which prior step it most directly continues

Parent Trace is a logical continuity relation. It is not a storage
locator and it is not an integrity proof.

---

# 7. Origin

The Origin field identifies the best known bounded reconstruction
entrypoint.

Origin MUST use an adapter-prefixed locator form:

```text
adapter-name://...
```

The adapter prefix identifies the reconstruction strategy, not the
continuity identity.

Examples include:

```text
git://github.com/Tiinex/.github@1e6762cc97ef151828fa7466ec22e7a2f1762909:.topics/rfc/001-1.trace.md
https://github.com/Tiinex/.github/blob/1e6762cc97ef151828fa7466ec22e7a2f1762909/.topics/rfc/001-1.trace.md
archive://rfc-snapshot.zip/.topics/rfc/001-1.trace.md
file://C:/continuity/rfc/001-1.trace.md
```

Unknown adapters do not invalidate the artifact. They only limit
direct reconstruction by runtimes that do not support them.

---

# 8. Compatible With

Compatible With expresses bounded reconstructability rather than hard
canonical runtime authority.

The field helps runtimes and humans decide:
- whether the artifact is likely interpretable
- whether reconstruction may be partial
- whether a different adapter or runtime is required

This field SHOULD remain compact and semantically meaningful.

---

# 9. Footer Integrity

The footer SHOULD be placed at the bottom of the artifact so that it
does not interrupt continuity orientation or narrative reading.

The footer is the appropriate location for:
- body checksums
- integrity policies
- compact validation anchors

The footer SHOULD be excluded from its own integrity calculation.

---

# 10. Deliberate Exclusions

The header SHOULD NOT begin with:
- opaque database identifiers
- serialization internals
- protocol dumps
- parser-specific state
- checksum-first ordering

Such fields may be useful elsewhere, but they do not carry the highest
orientation value for continuity reading.

---

# 11. Publication Direction

This document is written as a markdown continuity artifact, not as a
final Internet-Draft submission format.

For eventual publication, the content should be converted into the
required Internet-Draft authoring and stream process used by the RFC
Editor ecosystem.

The intended contribution of this iteration is to stabilize the model
before transport and publication mechanics are finalized.

---

# 12. Security Considerations

Origin fields, lineage references, and compatibility claims may expose
historical locations, infrastructure details, or operational context.

Implementations SHOULD support omission, redaction, or bounded
disclosure where appropriate.

---

# 13. IANA Considerations

This document has no IANA actions.

---

# 14. Conclusion

The discrimination-first continuity envelope preserves what matters
most at the start of reading:
- chain membership
- reconstruction entrypoint
- bounded compatibility
- recoverable orientation

By separating Parent Trace, Origin, and footer integrity into distinct
responsibilities, the model improves continuity portability without
overloading any one field.
